# Tellaro PM — Roadmap & Architecture

## Vision

An AI-orchestration-first project management platform. AI agents are the primary workforce — humans direct, review, and approve. The PM layer (boards, tasks, issues) is the **control plane** for coordinating AI agents. Chat is the **command interface**. GitHub is the **execution surface** where work materializes as branches, commits, and PRs.

**Core loop:**
1. Human (or AI) creates/refines work items
2. Work gets assigned to a user
3. That user's AI agents autonomously pick up, plan, execute, and deliver
4. Human reviews/approves or redirects
5. Repeat

## Domain Model

### Core Entities

| Entity | Description |
|---|---|
| **User** | A human with an account. Has one or more Agent Installations. Can be assigned work. Authenticates via pluggable providers: local (username/password), GitHub OAuth, or OIDC (Okta, Azure AD, Keycloak, etc.). Auth provider is resolved per email domain — admin configures domain→provider mappings in the UI. |
| **Agent Installation** | A Python daemon on a user's machine (e.g., "jhend-windows", "jhend-wsl"). Connects to backend via WebSocket. Manages one or more Claude Code processes. |
| **Agent Persona** | A named role/skill configuration for a Claude Code instance within an installation (e.g., "Project Manager", "Frontend Dev", "QA Reviewer"). Created and managed via the web UI, stored in OpenSearch, pushed to agents over WebSocket. Agents discover available Claude Code skills/profiles and report them to the backend for persona configuration. |
| **Project** | Top-level container linked to one or more GitHub org repos. |
| **Task** | A unit of work within a project. Can be synced from GitHub Issues or created natively. Has status, assignee, priority, and linked chat. |
| **Subtask** | Breakdown of a task. Can be auto-generated by an AI agent (e.g., PM persona breaks a feature into subtasks). |
| **Chat Session** | Exists per-project, per-task, per-subtask, or as a free-form session. Supports `@agent` and `@skill` mentions. Optional working directory binding. |
| **Work Request** | Cross-user delegation. User A requests User B's agent to perform work. Requires User B's approval. |
| **Skill Registry** | Catalog of discovered Claude Code skills and agent profiles available across all installations. |

### Relationships

```
User 1──* AgentInstallation 1──* AgentPersona
User 1──* TaskAssignment
Project 1──* Task 1──* Subtask
Project *──* GitHubRepo
Task/Subtask 1──1 ChatSession
Project 1──* ChatSession (project-level chat)
User 1──* ChatSession (free-form sessions)
ChatSession 1──* Message
AgentPersona ──* WorkExecution
User A ──WorkRequest──> User B (requires approval)
```

### Example Scenario

**User: jhend** has 2 Claude Code Max Pro subscriptions:
- **Agent: jhend-wsl** → 2 Claude instances:
  - Persona: "Project Manager" — plans features, breaks into subtasks, coordinates
  - Persona: "QA Reviewer" — reviews PRs, runs tests, checks quality
- **Agent: jhend-windows** → 2 Claude instances:
  - Persona: "Frontend Dev" — implements Vue.js components, pages, tests
  - Persona: "Backend Dev" — implements FastAPI endpoints, models, tests

**Workflow:**
1. jhend creates a task: "Implement user profile page"
2. In task chat: `@pm plan this feature`
3. PM persona on WSL agent picks it up, creates subtasks (API endpoint, Vue component, tests, etc.)
4. `@frontend-dev implement subtask #3 — profile page component`
5. Frontend Dev persona on Windows agent picks it up, works in the linked repo
6. Agent streams progress back to chat in real-time
7. Frontend Dev creates a PR, marks subtask done
8. `@qa review PR #45`
9. QA persona reviews, leaves comments, approves or requests changes

### Why OpenSearch as Primary Store

- Team already operates large-scale OpenSearch clusters with built-in redundancy
- Full-text search across all entities (tasks, chat messages, agent logs) out of the box
- Excellent for high-volume time-series data (chat, agent output streams, activity logs)
- Aggregations power dashboards and analytics without additional tooling
- Document model fits the domain well (tasks, messages, agent configs are naturally documents)
- Relational integrity handled at application layer with document references and validation

### OpenSearch Index Design

| Index | Purpose |
|---|---|
| `tellaro-users` | User profiles, preferences, OAuth tokens |
| `tellaro-projects` | Project metadata, linked repos, member lists |
| `tellaro-tasks` | Tasks and subtasks with status, assignee, priority |
| `tellaro-chat-messages` | All chat messages across all session types |
| `tellaro-chat-sessions` | Chat session metadata (scope, participants, working dir) |
| `tellaro-agents` | Agent installation registrations, capabilities, status |
| `tellaro-personas` | Persona/skill configurations |
| `tellaro-work-items` | Work dispatched to agents (queue, status, output ref) |
| `tellaro-work-requests` | Cross-user work delegation with approval status |
| `tellaro-activity` | Audit log / activity stream |
| `tellaro-github-sync` | GitHub webhook events and sync state |

### Authentication Flow

Provider-agnostic with domain-based routing, configurable in the UI:

1. **Bootstrap**: First-run local admin account for initial configuration (always available)
2. **Login flow**: User enters email → backend discovers provider for that email domain → redirects accordingly
3. **Supported providers**:
   - **Local**: Username/password with bcrypt hashing
   - **GitHub OAuth**: Redirects to GitHub (which may go through Okta if configured as GitHub's IdP)
   - **OIDC**: Generic OpenID Connect — supports Okta, Azure AD, Keycloak, Auth0, etc.
4. **Domain mapping**: Admin configures in UI which email domains use which provider (e.g., `@company.com` → OIDC/Okta, `@gmail.com` → GitHub OAuth, unconfigured → local)
5. Backend exchanges OAuth/OIDC code for user info, creates/updates user in OpenSearch
6. JWT session tokens for API access

## Phased Roadmap

### Phase 0 — Scaffolding & CI
- Monorepo structure: `frontend/`, `backend/`, `agent/`, `docs/`
- **Backend**: FastAPI + Ruff + Pyright + pytest + opensearch-py
- **Frontend**: Vue 3 + Vite + TypeScript + ESLint + Vitest + Playwright
- **Agent**: Python package structure + asyncio foundation
- Docker Compose: OpenSearch, Redis (local dev)
- GitHub Actions CI: lint, type-check, test for all three packages
- Pre-commit hooks (ruff, eslint, pyright)
- OpenSearch index templates and management scripts

### Phase 1 — Auth, Users & Agent Registration
- Local admin bootstrap flow
- GitHub OAuth integration (Okta-backed)
- User CRUD and profile management
- Agent Installation registration protocol (WebSocket handshake)
- Agent heartbeat / presence tracking
- Persona/skill discovery: agent scans Claude Code and reports capabilities
- Backend agent registry and status API
- **Tests**: auth flow, agent registration, presence

### Phase 2 — Projects, Tasks & GitHub Sync
- Project CRUD, link to GitHub org repos
- Task/subtask CRUD with status workflow (backlog → assigned → in_progress → review → done)
- GitHub Issues bidirectional sync
- PR status display on tasks
- Webhook receiver for GitHub events
- Assignment to users
- **Tests**: CRUD operations, GitHub sync, webhook handling

### Phase 3 — Chat System
- Chat session creation scoped to: project, task, subtask, or free-form
- Free-form sessions with optional working directory selection
- Real-time messaging via WebSocket
- Message persistence to OpenSearch
- `@agent` and `@skill` mention parsing
- Markdown rendering with code blocks
- Chat history, search (leveraging OpenSearch)
- Session management (create, archive, switch — similar to Claude Code web)
- **Tests**: messaging, mention parsing, search, session lifecycle

### Phase 4 — Work Dispatch & Agent Execution
- `@agent` mention in chat → Work Item created and dispatched
- Work queue per agent installation, per persona
- Agent picks up work items via WebSocket
- Agent spawns Claude Code subprocess with appropriate persona/skill config
- Real-time output streaming from Claude Code → backend → chat
- Work status lifecycle (queued → running → completed/failed)
- Output artifacts (files changed, PRs created) linked back to task
- **Tests**: dispatch flow, agent execution mock, status transitions

### Phase 5 — Multi-Instance Orchestration & Cross-User Delegation
- Single agent installation managing multiple concurrent Claude Code instances
- Persona-to-instance assignment and resource management
- Coordination: PM persona creates subtasks → other personas auto-pick up
- Cross-user Work Requests with approval workflow
- Notification when approval is needed
- Agent-to-agent coordination through the backend
- **Tests**: concurrent execution, approval workflow, orchestration

### Phase 6 — Frontend Buildout
- Dashboard: active projects, agent status overview, recent activity
- Project view: task board (Kanban), list view, Gantt-style timeline
- Task detail: description, subtasks, linked PRs, agent activity log
- Chat interface: embedded per-task, per-project, and standalone sessions
- Agent management: view installations, personas, status, output history
- Work request inbox: pending approvals
- Real-time updates everywhere via WebSocket
- **Tests**: E2E flows for all major user journeys

### Phase 7 — Polish & Operational Features
- Notifications (in-app, optional email/Slack webhook)
- Activity feed / audit log with full-text search
- Project analytics (velocity, agent utilization, task throughput)
- Skill registry browser (discover and install Claude profiles)
- Admin panel: org settings, user management, OpenSearch cluster health
- Dark mode, responsive design, accessibility
- Documentation: user guide, agent setup guide, API reference

## Design Decisions (Resolved)

1. **Agent persona config**: Managed via UI, stored in OpenSearch, pushed to agents over WebSocket. Agents are execution runtimes — no local persona config files needed. On initial install, the agent scans the local Claude Code installation to discover prebuilt agents (personas) and skills, then reports them to the backend for the user to configure/enable via UI.

2. **Hierarchical configuration loader**: Both backend and agent use the same config pattern. Resolution order (highest priority first):
   - **Docker/Kubernetes Secrets** — reads from `/run/secrets/{KEY}` files
   - **Environment Variables** — `{SECTION}_{KEY}` format (e.g., `OPENSEARCH_HOST`), loaded from `.env` via `python-dotenv`
   - **Default value** — hardcoded fallback

   Implementation uses a `get_default_setting(section, key, default)` function that walks this hierarchy. Settings class built on `pydantic-settings` `BaseSettings` with validators. Both `backend/` and `agent/` share this pattern. Local development uses `.env` files (see `.env.example` in each package). Every config value supports all override layers.

   CLI arguments are supported as the highest-priority override for the agent (since it runs as a user-facing daemon), making the full agent hierarchy: **CLI → Docker/K8s Secrets → ENV → Default**.

## Open Design Decisions

1. **Claude Code interface**: CLI subprocess (`claude --flags`)? Or the SDK if one becomes available?
2. **Agent auto-assignment**: When PM persona creates subtasks, should they auto-assign to other personas on the same user, or require explicit `@mention`?
3. **Chat session limits**: Max concurrent free-form sessions per user?
4. **OpenSearch versioning**: Minimum supported OpenSearch version for the deployment?
5. **Shared config package**: Extract `get_default_setting` + base settings into a `tellaro-common` package shared by backend and agent, or keep separate implementations?
